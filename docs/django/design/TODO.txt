8. 处理点赞 / 取消赞请求乱序问题
============================================================

解决方案: 使用短期缓存记录"预取消赞"状态 (pre-cancel like)

1) 当执行取消赞
   如果数据库中没有对应的点赞记录 (deleted_entries == 0), 说明可能发生了请求乱序
   将该"取消赞"操作写入缓存, 短时间内生效

   if deleted_entries = 0:
       cache.set('pre_cancel_like::tweet::{tweet_id}::user::{user_id}', True, timeout=1)
       {user_id} precancel {tweet_id}, 1s 内有效

2) 当执行点赞时
   检查是否存在对应的"预取消赞"缓存;
   如果存在, 说明之前收到过取消赞, 当前点赞应被忽略(不执行) + 同时删除该缓存

   def create():
       cache_key = f'pre_cancel_like::tweet::{tweet_id}::user::{user_id}'
       if cache.get(cache_key):
           cache.delete(cache_key)
           return  # 忽略本次点赞操作
       # 正常执行点赞逻辑

10. Tweet & Newsfeed Redis 缓存设计
============================================================
Tweet API 接口设计: 获取推文, 创建推文, 不支持删除推文 和 修改推文

缓存结构:
  Memcached  key: 'tweet:{tweet_id}'        value: 序列化的单条 Tweet 对象 (点查)
  Redis      key: 'user_tweets:{user_id}'   value: 该用户发的所有 tweet 列表
  Redis      key: 'user_newsfeeds:{user_id} value: 该用户的 newsfeed 列表


1) Tweet Redis 缓存方案对比
----------------------------------------------------------------
方法 1: 只缓存 id 列表
  value: [tweet_id_1, tweet_id_2, ...]
  优点: 内存占用小; 删除/编辑只需动 Memcached, 无需更新 id 列表
  缺点: 读取需要二次查询

方法 2: 缓存完整序列化对象 (当前方案)
  value: [序列化的 tweet 对象, ...]
  优点: 读取无需二次查询，性能最优; 实现简单
  缺点: 内存占用大; 删除/编辑 tweet 时必须同步更新缓存，否则脏数据

方法 3: 方法 1 + Memcached 双层组合缓存
  创建 tweet : 写 DB → 写 Memcached → lpush Redis id 列表
  删除 tweet : 软删除 DB → 删除 Memcached (无需动 id 列表)
  编辑 tweet : 更新 DB → 删除 Memcached (无需动 id 列表)
  查看 tweet : Redis 取 id 列表
               → Memcached.get_many(ids) 批量取
               → cache miss: Tweet.objects.filter(id__in=[...]) 补查 DB


2) Newsfeed Redis 缓存方案对比
----------------------------------------------------------------
方案 1: 只缓存 id 列表 + 软删除 (has_deleted)
  删除 tweet  → 标记 has_deleted=True → 删除 Memcached (无需动 id 列表)
  读取列表    → 跳过 has_deleted=True 的条目

方案 2: 缓存完整序列化对象 (不推荐)
  删除/修改 tweet 时需要:
  1) fanout 逐一更新所有相关用户的缓存
  2) 在 Redis list 中定位并替换该 tweet (list 不支持按值查找，几乎无法实现)
     → 这也是部分网站不支持编辑已发布内容的原因


3) 取关后的 Newsfeed 处理
----------------------------------------------------------------
场景:
  1. A 关注 B
  2. B 发 tweet1 → fanout → NewsFeed(user=A, tweet=tweet1)
  3. A 取关 B

方法1: 取关时主动清理 (写时处理)
  取关(重): 批量标记 NewsFeed is_deleted=True
  读取(轻): 无额外逻辑

方法2: 读取时过滤 (读时处理)
  取关(轻): 无额外逻辑
  读取(重): 每次需要额外过滤

  Twitter / 微博 选方法1: 用低频操作的额外代价，换取高频操作的性能
  取关是极低频操作 (一天取关几次), 代价高一点可以接受
  读取是极高频操作 (一天刷几百次), 必须保证轻量快速

15. 优化"下拉刷新"
============================================================
1) 下拉刷新: 简单起见, 下拉刷新不做翻页机制, 直接加载[所有]新数据
   长期未刷新, 新帖子过多: 不采用下拉刷新, 而是重新加载首页 (例: 仅显示最新 20 条, 不与旧帖子拼接)

2) 如何处理翻页结果中的无权限内容?
   分页权限过滤策略: 先 queryset 取全量数据, 再在内存中按黑/白名单过滤 (黑/白名单 写入缓存加速读取)
                  而非在 queryset 层直接加权限条件 (会导致分页数量不准且查询效率低)
   例: page_size=3，用户A无权查看帖子9和帖子4
       第一次查询 → [9, 8, 7]，过滤9 → 返回 [8, 7]  (不足 page_size)
       第二次查询 → [6, 5, 4]，过滤4 → 返回 [6, 5]
       最终拼接   → [8, 7, 6, 5, ...]  (需多次补充查询才能凑满一页)

20. 缓存 tweet_photo (消除 get_photo_urls 的 N+1)
============================================================
现状: TweetSerializer.get_photo_urls 用 obj.tweetphoto_set.all() 取图,
     即使 Tweet 本体已从 Redis 取到, photo 仍每条一次 DB —— 一页 20 条 = 20 次查询 (N+1)
     (feed 是最热路径, 这是当前最大的低效点)

前提: 本项目 tweet_photo 不可变 —— 仅发推时 create_photos_from_files 创建, 无 "编辑照片" 入口

方案 A (推荐): 反范式到 Tweet
  Tweet 加 photo_urls 字段 (JSON/array), 创建时写入
  读: Redis 取 Tweet blob 自带 photo_urls => 0 次额外查询
  与现有 "list 缓存 = 单 model 序列化" 结构天然融合, 读路径零改动

方案 B: Memcached 缓存 photo 列表 (仿 cached_user)
  按 tweet_id 缓存 photo 列表, get_photo_urls 走 Memcached
  读: 每条 1 次 memcached (无 DB); 多一层 round-trip + 需失效逻辑

⚠️ 坑:
  1. 若 photo.file.url 是 S3 presigned (会过期) URL, 缓存里只能存 key/路径, URL 读时现拼
  2. "不可变" 是前提; 若将来支持编辑照片, 需在增删 photo 时同步刷新 Tweet 缓存

21. 缓存穿透 (cache penetration): retrieve 查不存在的 tweet
============================================================
现状: retrieve 走 get_object_through_cache —— cache miss 才回源 DB, 但 "不存在的 id"
     从不写缓存, 于是对 /api/tweets/-1/ 反复请求每次都 cache miss -> 打 DB
     恶意刷不存在的 id => DB 被击穿 (cache penetration)

区分三个 "穿":
  穿透 penetration: 查 "不存在" 的数据, 缓存永远不命中, 每次落 DB   <- 本条
  击穿 breakdown:   单个热点 key 过期瞬间, 大量请求同时回源       (已用 load_objects 的锁缓解)
  雪崩 avalanche:   大量 key 同时过期, 集体回源                  (TTL 加随机抖动缓解)

方案:
  A. 缓存空值: DB 查不到时也写一个 "空标记" (短 TTL, 如 60s), 后续请求直接被挡在缓存层
     cache.set(key, EMPTY, ex=60)   下次 get 命中 EMPTY -> 直接 404, 不落 DB
     代价: 占少量内存; 真数据创建后需等空标记过期或主动删除
  B. 布隆过滤器 (Bloom Filter): 请求前先判 id 是否 "可能存在", 不存在直接拒
     适合海量 id、空值缓存装不下的场景; 实现复杂, 本项目规模用 A 即可

⚠️ A 的空值需可与真实对象区分 (现在 get_object_through_cache 的 `if obj:` 把 None 当 miss,
   需引入一个明确的 EMPTY 哨兵值, 否则空标记会被当成 cache miss)

22. LLM 内容审核: 缓存失效 + fanout 时序漏洞 + 读时过滤
============================================================
现状: moderate_tweet (Celery task) 调 Claude API 判完 SAFE/FLAGGED 后只 .update() DB,
     缓存和展示层完全不知情, moderation_status 形同虚设 (先简化实现, 只跑通 LLM API 调用本身)

遗留问题 (按依赖顺序):

1) 两层缓存不会自动感知 moderation_status 的变化
   - Memcached (tweet:{tweet_id} 单点缓存): .update() 走 QuerySet, 不触发 post_save 信号,
     缓存里的 moderation_status 停留在发帖那一刻 (PENDING), 需要手动 invalidate_cached_object
   - Redis (user_tweets:{user_id} 列表缓存): 缓存的是发帖时序列化的完整 Tweet 快照,
     PENDING 状态被永久固化进去, 光 invalidate Memcached 不够, 得把这个 key 也删掉触发重建
   - newsfeed 路径反而天然没事: NewsFeed 在 Redis 里只存 (user_id, tweet_id, created_at),
     真正内容通过 cached_tweet -> Memcached 单点缓存现读, 只要 1) 修好, 这条路径就跟着对

2) fanout 已经跑在审核完成之前, 无法靠调整顺序解决
   views.py 里 fanout_to_followers() 和 moderate_tweet.delay() 前后脚派发,
   但 fanout 只写 DB (几十毫秒), moderate_tweet 要等外部 API 网络往返 (几百毫秒到几秒)
   等判定 FLAGGED 时, 这条推文早已出现在所有粉丝的 NewsFeed 表里
   => 结论: 不能指望"审核完成前不让扩散", 只要审核是异步的, 这个竞态窗口就消不掉

3) 因此正确的修法是读时过滤, 不是重新设计 fanout
   在 TweetViewSet.list/retrieve、NewsFeedViewSet.list 序列化前排除 moderation_status == FLAGGED
   的推文 (1) 修好后, 排除判断不需要额外查询, 直接读 cached_tweet.moderation_status 即可)
   这与真实系统 (Twitter 等) 的做法一致: 先发布, 异步审核跟上, 一旦违规就在读路径下架,
   而不是试图撤销已经派发出去的 fanout 记录
